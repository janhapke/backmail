## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Test input → parseImapUrl() | Test suite provides untrusted URL strings (invalid protocols, missing credentials, malformed) |
| ImapFlow client mock → Unit tests | Mocked client may simulate connection failures, permission errors, or bad IMAP responses |
| Temporary repo → Integration tests | Integration tests read .eml files from temp filesystem (controlled by test setup) |
| CLI argv → restore subcommand | User provides target URL with embedded credentials; test verifies credentials not leaked to logs/errors |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-5-01 | Tampering | URL parsing (test input validation) | mitigate | Unit tests verify parseImapUrl() rejects invalid protocols, missing credentials, malformed URLs; use Node.js URL constructor in implementation (safe parsing) |
| T-5-02 | Information Disclosure | Error message output in CLI tests | mitigate | CLI tests verify that error messages do NOT include password-embedded URL; implementation will sanitize errors per Pitfall 4 |
| T-5-03 | Information Disclosure | Mock ImapFlow in unit tests | accept | Mocks are test-only; no real IMAP operations in unit tests; acceptable risk |
| T-5-04 | Denial of Service | Integration test setup/teardown | mitigate | Tests use temporary directories and cleanup in afterAll(); no orphaned resources left after failures |

</threat_model>
