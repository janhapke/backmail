# Milestone v1.1: Repository-Centric UX

**Status:** ✅ SHIPPED 2026-05-04
**Phases:** 6–9
**Total Plans:** 9

## Overview

v1.1 replaced the central config model (`~/.config/backmail/config.json`) with self-contained backmail repositories. The build order followed the research recommendation: credential/config infrastructure first, then repository discovery, then migrating existing commands, then the `init` command that ties it all together.

## Milestone Summary

**Key Decisions:**

- Central `~/.config/backmail/config.json` replaced by per-repo `.backmail/` (self-contained, git-like)
- `.backmail/` as repository marker (walk-up detection, git muscle memory)
- `archive/` subdirectory holds the git repo (keeps `.backmail/` outside version control)
- `passwordRef` format: `keyring:service=X;account=Y` (self-documenting, extensible to `env:VAR`)
- `process.stdin.isTTY === true` for TTY detection (`undefined` treated as non-TTY — D-10)
- REPO-04 guard checks `.backmail/` existence BEFORE collecting any prompts

**Issues Resolved:**

- Legacy account registry (`resolveAccount`, `LegacyAccountConfig`, `--account` flag) fully removed
- Central config model replaced — no backward compatibility shims needed

**Issues Deferred:**

- CR-01: Unsafe BigInt() from corrupted folder JSON state (deferred from v1.0, still deferred)
- restoreAccount() dead `config` parameter — redundant loadRepositoryConfig() call on every restore

**Technical Debt Incurred:**

- Phases 6, 7, 8 have no VALIDATION.md (predate Nyquist workflow)
- Phase 9 VALIDATION.md: nyquist_compliant=false (2 manual-only items: TTY prompts, OS keyring — architectural limitation)
